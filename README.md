## 契约优先级

发生描述冲突时按以下规则处理：

1. 产品是否属于 MVP，以 PRD 为准。
2. HTTP 路径、字段、状态码、签名和幂等，以《AI服务接口文档》为准。
3. 禅道字段、原生必填、反馈枚举、Hook 和内容哈希，以《禅道 22.3 集成与字段映射文档》为准。
4. 提交校验策略和上线门槛，以《Bug 与反馈提交完整性校验设计文档》为准。
5. 向量、切片和帮助知识发布，以《禅道 Bug 与文档向量同步实现文档》为准。
6. 客户展示流程只用于沟通，不作为开发接口契约。

## 首批 MVP

包含：

- Bug、文档同步和文本向量索引。
- Bug 相似检索和 AI 分析。
- 单文档摘要与问答。
- 登录后业务页面的全局 AI 客服。
- `help`、`bug`、`doc`、`feedback` 四种会话范围。
- 反馈详情页实时白名单取数，不要求反馈全文向量化。
- Bug、反馈提交完整性校验。
- 对象级权限回调、双向 HMAC、租户隔离、审计和人工确认。
- 平台帮助知识 Manifest、审核、导入和版本过滤。

不包含：

- 反馈自动分类和生成 Bug/需求草稿。
- 项目范围跨对象问答和项目风险周报。
- 通用跨类型推荐。
- 图片 OCR/视觉理解。
- 模型管理后台和客户自助上传帮助资料。

## 固定技术基线

- 禅道：开源版 22.3，研发视界。
- 禅道扩展：`extension/custom`，不修改核心文件。
- AI API：`/api/v1`。
- 禅道 Callback：`/api/ai/v1`。
- 双向调用：两个独立 HMAC Client 和 Secret。
- AI 服务：FastAPI、PostgreSQL、Redis/Celery、Qdrant、OpenAI Compatible 模型网关。
- 提交完整性校验：先 `observe/warn`，达到量化门槛后再启用 `enforce`。

## 开发前仍需外部提供

- 模型 `base_url`、模型名、Embedding 维度、额度和密钥。
- 数据是否允许出网及脱敏规则。
- 审核通过的平台帮助 Markdown/FAQ。
- 客户禅道实际版本和自定义字段清单。
- 可联调环境、测试账号和冻结测试数据集。

以上外部输入不影响搭建工程骨架和 Mock 联调，但会阻塞真实模型、客户字段和最终验收。
