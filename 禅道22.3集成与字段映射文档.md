# 禅道 22.3 集成与字段映射文档

版本：v1.0
日期：2026-07-10
文档类型：禅道扩展集成基线
适用范围：禅道开源版 22.3、研发视界、Bug、反馈、文档、全局 AI 客服、提交完整性校验

## 1. 基线与目标

首批开发以当前工作区禅道 `VERSION=22.3` 为唯一兼容基线。客户环境版本不同不得直接假设兼容，必须先执行扩展加载、页面 DOM/ZIN 查询和字段映射回归测试。

本文用于固定：

- 禅道 22.3 原生字段与 AI 服务规范字段的映射。
- 原生必填规则与 AI 质量规则的边界。
- Bug、反馈创建页提交前校验的接入方式。
- Bug、文档、反馈详情页和全局 Footer 的扩展入口。
- 反馈实时取数的字段白名单。

代码依据：

```text
VERSION
module/bug/config.php
module/bug/config/form.php
module/bug/ui/common.field.php
module/bug/ui/create.field.php
module/bug/ui/create.html.php
module/feedback/config.php
module/feedback/ui/create.html.php
module/feedback/model.php
module/common/view/footer.html.php
```

## 2. 扩展原则

- 使用 `extension/custom`，不直接修改禅道核心文件。
- 浏览器只调用禅道同源 Controller，AI 服务地址和 HMAC Secret 不下发前端。
- 前端字段提示用于改善体验，最终必填、权限、内容哈希、校验有效期和强制提交权限必须在禅道服务端重新确认。
- 保留禅道原 CSRF、权限、文件上传、工作流字段和 `form::data` 处理逻辑。
- AI 服务不可用时按文档定义降级，不能造成表单丢失或绕过禅道原生必填规则。
- 自定义字段从禅道实际工作流/字段配置动态读取，不在 PHP 或 AI Prompt 中写死客户字段 ID。

## 3. 页面和扩展入口

| 场景 | 禅道模块/方法 | 建议扩展位置 | 说明 |
| --- | --- | --- | --- |
| 全局机器人 | `common/footer` | `extension/custom/common/ext/view/footer.*.hook.php` | 登录后的业务页面统一加载；登录、安装、升级、公开页服务端排除 |
| Bug 详情 AI 面板 | `bug/view` | `extension/custom/bug/ext/ui/view.*.hook.php` | 当前仓库已有最小 Hook，可继续演进 |
| 文档详情 AI 助手 | `doc/view` | `extension/custom/doc/ext/ui/view.*.hook.php` | 绑定当前文档 ID |
| 反馈详情 AI 客服 | `feedback/view` | `extension/custom/feedback/ext/ui/view.*.hook.php` | 绑定当前反馈 ID，通过内部取数接口实时读取 |
| Bug 创建校验 | `bug/create` | `extension/custom/bug/ext/js/create/` + 服务端扩展 | JS 拦截首次提交；服务端重新计算字段和哈希 |
| 反馈创建校验 | `feedback/create` | `extension/custom/feedback/ext/js/create/` + 服务端扩展 | 保留原文件上传与通知逻辑 |

具体 Hook 文件名必须先用最小无业务逻辑扩展验证一次加载顺序；验证结果写入自动化烟测，避免禅道升级后静默失效。

## 4. Bug 创建字段映射

### 4.1 原生字段

禅道 22.3 Bug 创建表单和 Model 主要字段：

| 禅道字段 | AI 规范字段 | 类型 | 处理方式 |
| --- | --- | --- | --- |
| 路由 `productID` / `product` | `product_id` | ID | 禅道服务端从当前路由和表单确认 |
| `branch` | `branch_id` | ID | 分支产品时使用 |
| `module` | `module_id` | ID | 页面原生要求选择模块；服务端按实际配置处理 |
| `project` | `project_id` | ID | 可选上下文 |
| `execution` | `execution_id` | ID | 可选上下文 |
| `openedBuild[]` | `opened_build_ids` | ID 数组 | 原生必填，规范化排序后参与内容哈希 |
| `title` | `title` | string | 原生必填，Trim 后参与校验 |
| `steps` | `raw_steps_html` | HTML | 清洗危险标签，保留原值供禅道提交 |
| `steps` 中 `[步骤]` 段 | `repro_steps` | string | 解析后的语义字段，不回写覆盖原文 |
| `steps` 中 `[结果]` 段 | `actual_result` | string | 解析后的语义字段 |
| `steps` 中 `[期望]` 段 | `expected_result` | string | 解析后的语义字段 |
| `type` | `bug_type` | enum | 使用禅道原枚举值 |
| `severity` | `severity` | integer | 默认 3，检查与描述是否明显矛盾 |
| `pri` | `priority` | integer | 默认 3，AI 只提示不自动修改 |
| `os` | `operating_system` | string/list | 可选环境信息 |
| `browser` | `browser` | string/list | 可选环境信息 |
| `hardware` | `hardware` | string | 可能由自定义字段/配置展示，存在时映射 |
| `assignedTo` | `assigned_to` | account | 不用于内容完整性硬阻断 |
| `deadline` | `deadline` | date | 不用于内容完整性硬阻断 |
| `feedbackBy` | `reporter_name` | string | 可选 |
| `notifyEmail` | `notify_email` | string | 只做格式和敏感信息检查 |
| `keywords` | `keywords` | string | 可选 |
| 上传文件元数据 | `attachments` | array | 只传临时 ID、文件名、MIME、大小，不传未提交附件正文 |

### 4.2 原生必填与 AI 规则

禅道 22.3 Model 原生必填字段是：

```text
title, openedBuild
```

页面同时依赖产品上下文，并将模块、严重程度、优先级作为重要输入。实现时：

- 原生必填失败直接使用禅道错误，不调用 AI。
- AI 固定规则额外检查标题信息量、`steps` 是否仍是空模板、三个语义段是否缺失、乱码和重复内容。
- MVP `warn` 模式下，AI 认为 `repro_steps/actual_result/expected_result` 不充分时只给警告；经过样本验收后才允许配置为 `enforce`。
- `steps` 模板标记支持当前语言：中文 `[步骤]/[结果]/[期望]`，英文 `[Steps]/[Results]/[Expectations]`；解析失败时保留全文并返回结构补充建议。
- 用户确认 AI 建议不代表允许自动改写 `steps`；如后续支持一键整理，必须显示差异并由用户确认。

### 4.3 Bug 类型枚举

首批透传禅道原值，例如：

```text
codeerror, config, install, security, performance,
standard, automation, designdefect, others
```

规则模板按原值匹配，不使用展示中文作为稳定枚举。

| --- | --- | --- | --- |
| `product` | `product_id` | ID | 原生必填 |
| `module` | `module_id` | ID | 可选 |

| `pri` | `priority` | integer | 默认值按禅道配置 |
| `title` | `title` | string | 原生必填 |
| `public` | `is_public` | boolean | 只用于权限/展示，不作为质量分数 |
| `desc` | `description_html` | HTML | 清洗后生成纯文本用于规则和模型 |
| `assignedTo` | `assigned_to` | account | 不用于内容完整性硬阻断 |
| `feedbackBy` | `reporter_name` | string | 可选，可能包含客户联系人 |
| `source` | `source` | string | 可选；客户启用后建议纳入模板 |

| 上传文件元数据 | `attachments` | array | 与 Bug 相同，只传元数据 |

### 5.2 原生必填与 AI 规则

禅道 22.3 Model 原生必填字段是：

```text
product, title
```

AI 质量校验默认要求 `description_html` 去标签后的正文具有可处理信息。MVP 规则：

- `desc` 为空或仅包含空标签时，固定规则给出 `warning`；试点校准后可按反馈类型提升为硬规则。
- `title` 与 `desc` 完全相同、纯占位、乱码或重复字符时返回问题。
- 客户名称、固件版本、设备型号并非禅道 22.3 标准反馈字段；客户要求必填时必须通过工作流自定义字段或统一描述模板提供，不能假设原生存在。
- 自定义字段使用 `custom_fields` 规范对象传给 AI，字段代码和可见性由禅道服务端白名单决定。

### 5.3 反馈类型映射

禅道 22.3 原生反馈类型：

| 原值 | 中文 | AI 校验模板 |
| --- | --- | --- |
| `requirement` | 需求 | 功能场景、当前痛点、期望效果、影响用户 |
| `bug` | Bug | 发生场景、现象、环境版本、影响范围、复现信息 |
| `task` | 任务 | 目标、范围、期望完成结果、时间要求 |
| `question` | 问题 | 具体问题、产品版本、已尝试方法、期望帮助 |
| `advice` | 建议 | 使用场景、改进点、预期收益 |
| `other` | 其他 | 背景、具体内容、预期处理结果 |

“投诉”“体验问题”等类型不是 22.3 原生枚举；如客户需要，先扩展字段/枚举并更新模板版本，再启用对应规则。

## 6. 反馈详情实时取数 Profile

`POST /api/ai/v1/source-objects/batch-get` 的 `field_profile=ai_chat` 对反馈只允许返回：

```text
id, product, module, type, pri, title, desc, public,
feedbackBy, source, status, openedBy, openedDate,
assignedTo, editedDate, repeatFeedback
```

规则：

- 禅道服务端先检查 `feedback/view` 功能权限、产品访问权限、公开/私有范围和删除状态。
- `notifyEmail`、`mailto` 默认不返回模型；确有业务需要时先脱敏并修改 Profile 版本。
- 附件只返回文件 ID 和安全元数据，正文通过独立附件接口按权限读取。
- 关联对象只返回类型和 ID；AI 服务检索或展示前再次逐个鉴权。
- 返回 `source_revision`，首选 `editedDate`，未编辑时使用 `openedDate`。
- Profile 变更必须升级 `field_profile_version` 并增加契约测试。

## 7. 提交校验接入流程

```text
用户第一次点击提交
  ↓
禅道原生前端校验
  ↓
禅道服务端规范化实际字段并计算 content_hash
  ↓
禅道服务端签名调用 submission-validations
  ↓
返回 pass / warn / block / degraded
  ↓
页面定位字段、确认或返回修改
  ↓
最终提交时禅道服务端重新规范化并计算 content_hash
  ↓
校验有效且权限满足后调用原创建逻辑
```

状态保存：

- `validation_id` 可以放入隐藏字段用于回传，但不能作为信任依据。
- 禅道服务端使用当前 Session/缓存保存 `user + form_instance_id + validation_id + content_hash + expires_at`。
- 最终提交时以服务端重新计算的哈希和服务端缓存为准，拒绝浏览器伪造 `pass`、角色或强制提交原因。
- 文件选择变化、自定义字段变化、模板版本变化都会使旧校验失效。
- 创建对象和保存 Outcome Outbox 在同一数据库事务中完成，随后可靠上报 AI 服务。

## 8. 内容哈希规范

- 按模板声明的字段顺序构建规范 JSON 对象，不使用浏览器表单字段自然顺序。
- 字符串统一 UTF-8、去除 CR、将 CRLF 转为 LF；只对声明为 Trim 的字段去除首尾空白。
- ID 转为十进制字符串；布尔值使用 JSON `true/false`；数组按字段语义排序，步骤文本等有序数组不得排序。
- HTML 在哈希时使用禅道服务端接收的原始规范值，AI 分析可以使用另外生成的清洗纯文本。
- 附件按 `client_file_id + file_name + mime_type + size` 排序后加入规范对象。
- 使用无额外空格、UTF-8、不转义 Unicode 的 Canonical JSON 计算 SHA-256。
- PHP 与 Python 必须共享固定测试样本，保证同一表单得到相同哈希。

## 9. 权限映射

| AI 动作 | 禅道权限基础 |
| --- | --- |
| 校验 Bug 草稿 | 当前用户拥有 `bug/create` 且可访问目标产品 |
| 校验反馈草稿 | 当前用户拥有 `feedback/create` 且可访问目标产品 |
| 强制提交 Block | 原创建权限 + 租户配置的 Override 角色 |
| Bug 上下文聊天 | `bug/view` + 产品/项目/执行对象权限 |
| 文档上下文聊天 | `doc/view` + 文档库和文档 ACL |
| 反馈上下文聊天 | `feedback/view` + 产品和公开/私有范围 |

Override 角色不能替代禅道原创建权限，也不能绕过 CSRF、产品访问和原生必填字段。

## 10. 兼容性与验收

开发前烟测：

1. `VERSION` 精确为 `22.3`。
2. 全局 Footer Hook 在已登录业务页加载，在登录/安装/升级页不加载。
3. Bug、文档、反馈 View Hook 能取得当前对象 ID。
4. Bug、反馈 Create 页面脚本只绑定一次，不造成重复提交。
5. 原生表单无 AI 服务时仍可按降级策略提交。

字段契约测试：

- Bug `title/openedBuild` 原生必填保持不变。
- Bug `steps` 中文和英文模板均能拆分；无法拆分时不丢失原文。
- 反馈六种原生类型映射到正确校验模板。
- 自定义字段只按白名单传输，隐藏字段和无权限字段不进入 AI 请求。
- 表单修改、附件变化和模板升级后旧内容哈希失效。
- 强制提交不能绕过原创建权限，原因和操作人进入审计。

升级策略：

- 客户禅道版本不是 22.3 时，先复制本文件形成新版本映射，不直接修改 22.3 基线。
- 每次禅道升级执行 Hook 加载、字段清单、权限、提交哈希和页面自动化回归。
- 兼容性未通过时关闭 AI 扩展，不影响禅道核心功能。
